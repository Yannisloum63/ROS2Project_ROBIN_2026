	\section{Introduction}
	Ce projet a pour objectif de realiser un robot aspirateur autonome en simulation en s'appuyant sur les chapitres 7 a 10 de \emph{Programming Robots with ROS}. Le cadre pedagogique initial etait ROS1. Nous avons choisi de relever le defi d'une implementation complete en ROS2 Humble.

	Les objectifs fonctionnels sont:
	\begin{enumerate}
		\item Evitement reactif d'obstacles (chapitres 7-8)
		\item Cartographie SLAM d'un environnement inconnu (chapitre 9)
		\item Navigation sur carte avec Nav2 (chapitre 10)
		\item Couverture de zone type aspirateur
	\end{enumerate}

	\section{Contexte et migration ROS1 vers ROS2}
	Le passage ROS1 vers ROS2 a impose des adaptations techniques importantes:
	\begin{itemize}
		\item \texttt{rospy} vers \texttt{rclpy}
		\item \texttt{catkin} vers \texttt{colcon/ament}
		\item \texttt{move\_base} vers Nav2
		\item \texttt{gmapping} vers \texttt{slam\_toolbox}
	\end{itemize}

	Cette migration a ete un axe fort du projet. Elle permet de garder la logique des chapitres du livre tout en utilisant une stack moderne et maintenue.

	\subsection{Materiel et logiciel}

		\item VM: Ubuntu 22.04 (VirtualBox)
		\item ROS: ROS2 Humble
		\item Simulation: Gazebo + TurtleBot3 Burger

	\subsection{Paquets utilises}
	\begin{itemize}
		\item \texttt{reactive\_avoidance} (developpe)
		\item \texttt{coverage\_planner} (developpe)
		\item \texttt{nav2\_bringup}, \texttt{navigation2}
		\item \texttt{slam\_toolbox}
	\end{itemize}

	\section{Architecture implementee}
	\subsection{Noeud reactive\_avoidance}
	Le noeud lit \texttt{/scan} et publie \texttt{/cmd\_vel}. Une version initiale binaire (avance/rotation) a ete stabilisee avec:
	\begin{itemize}
		\item hysteresis sur la distance de decision
		\item boucle de commande a frequence fixe
		\item zone prudente + zone urgence pour limiter les impacts
	\end{itemize}

	\subsection{Noeud coverage\_planner}
	Le noeud de couverture:
	\begin{itemize}
		\item lit \texttt{/map}
		\item genere une trajectoire de type lawnmower
		\item publie des objectifs sur \texttt{/goal\_pose}
		\item suit l'avancement via \texttt{/amcl\_pose}
	\end{itemize}

	\subsection{Configuration}
	Les fichiers de configuration principaux sont:
	\begin{itemize}
		\item \texttt{config/slam\_params.yaml}
		\item \texttt{config/nav2\_params.yaml}
	\end{itemize}

	\section{Methodologie de validation}
	\subsection{Validation par chapitre}
	Nous avons adopte une validation incrementalement stricte:
	\begin{enumerate}
		\item Chapitres 7-8: evitement reactif sans Nav2
		\item Chapitre 9: cartographie SLAM + sauvegarde carte
		\item Chapitre 10: localisation AMCL + navigation Nav2
		\item Couverture: envoi sequentiel d'objectifs sur carte
	\end{enumerate}

	\subsection{Metriques suivies}
	\begin{itemize}
		\item Couverture (\%)
		\item Temps de mission
		\item Nombre de collisions
		\item Distance parcourue
		\item Reexploration
	\end{itemize}

	\subsection{Preuves deja obtenues}
	Exemple de sortie de commande observee pendant les tests:
	\begin{bash}
	ros2 topic echo /cmd_vel
	linear:
		x: 0.2
	angular:
		z: 0.0
	---
	\end{bash}

	Cette preuve confirme la commande d'avance en zone libre. La preuve complementaire attendue pour valider pleinement les chapitres 7-8 est la commande de rotation proche obstacle (\texttt{linear.x = 0}, \texttt{angular.z != 0}).

	\section{Obstacles rencontres et resolutions}
	\subsection{Problemes d'environnement ROS2}
	\begin{itemize}
		\item \textbf{Symptome:} \texttt{Package not found} pour \texttt{nav2\_bringup} et paquets projet.
		\item \textbf{Cause:} workspace non source ou dependances non installees.
		\item \textbf{Resolution:} installation des paquets requis, build \texttt{colcon}, puis \texttt{source /opt/ros/humble/setup.bash} et \texttt{source ~/ros2\_ws/install/setup.bash}.
	\end{itemize}

	\subsection{Problemes de configuration Nav2}
	\begin{itemize}
		\item \textbf{Symptome:} \texttt{yaml\_filename is not initialized}, \texttt{No critics defined for FollowPath}.
		\item \textbf{Cause:} fichier \texttt{nav2\_params.yaml} initial trop minimal.
		\item \textbf{Resolution:} parametrage complet des plugins et critiques DWB, plus reglages des frames.
	\end{itemize}

	\subsection{Problemes TF et localisation}
	\begin{itemize}
		\item \textbf{Symptome:} \texttt{Invalid frame ID map/odom/base\_link}.
		\item \textbf{Cause:} incoherence \texttt{base\_link} versus \texttt{base\_footprint} et initialisation AMCL incomplete.
		\item \textbf{Resolution:} harmonisation des frames dans Nav2 et procedure explicite \texttt{2D Pose Estimate} dans RViz2.
	\end{itemize}

	\subsection{Problemes de stabilite physique en simulation}
	\begin{itemize}
		\item \textbf{Symptome:} robot qui percute un mur, tombe, et ne se releve pas.
		\item \textbf{Cause:} vitesses trop agressives et collisions trop violentes en VM.
		\item \textbf{Resolution:} reduction des vitesses, ajout d'hysteresis et de zones prudente/urgence dans le controle reactif.
	\end{itemize}

	\section{Discussion}
	Le choix ROS2 apporte une plus-value pedagogique mais introduit une complexite de mise au point plus elevee qu'en ROS1 (ecosysteme, Nav2, parametrage TF et lifecycle nodes). La virtualisation accentue aussi les instabilites physiques en simulation.

	Malgre ces difficultes, l'architecture du projet est maintenant solide et modulaire. Le travail restant est principalement de la validation experimentale et du tuning.

	\section{Plan de travail jusqu'a la version finale}
	\begin{enumerate}

		\item Valider chapitre 9 (carte exploitable sauvegardee)

		\item Lancer la campagne de metriques pour la couverture

	\end{enumerate}

	\begin{bash}
	export PROJECT_ROOT=$HOME/ros2_ws/src/ROS2Project_ROBIN_2026

		slam_params_file:=$PROJECT_ROOT/config/slam_params.yaml \
		use_sim_time:=true

	\end{bash}

		params_file:=$PROJECT_ROOT/config/nav2_params.yaml
	\end{bash}
